## 摘要
MDLM通过对预分配的掩码响应画布进行去噪来生成文本，这使得响应长度建模成为指令微调的核心。现有的MDLM常继承自回归惯例，在指令微调期间使用重复的\texttt{[EOS]}标记进行填充，赋予\texttt{[EOS]}作为语义终止符和填充令牌的双重角色。我们证明这种双重角色是大块解码下\texttt{[EOS]}溢出的根本原因。为了解耦这些角色，我们提出VoidPadding，它引入\texttt{[VOID]}用于填充，并保留\texttt{[EOS]}用于终止。在推理过程中，学习到的\texttt{[EOS]}信号实现早停，而学习到的\texttt{[VOID]}信号指导自适应响应画布扩展。在Dream-7B-Instruct上，VoidPadding在数学推理和代码生成基准测试中，将块大小平均的四任务均值比原始模型提高了\(+17.84\)个点，比RainbowPadding提高了\(+6.95\)个点，同时将解码NFE平均降低了55.7\%。代码见https://github.com/Haru-LCY/VoidPadding。

## Abstract
MDLMs generate text by denoising a preallocated masked response canvas, making response-length modeling central to instruction tuning. Existing MDLMs often inherit the autoregressive convention of using repeated \texttt{[EOS]} tokens for padding during instruction tuning, giving \texttt{[EOS]} a dual role as both a semantic terminator and a padding token. We show that this dual role is a root cause of \texttt{[EOS]} overflow under large-block decoding. To decouple these roles, we propose VoidPadding, which introduces \texttt{[VOID]} for padding and reserves \texttt{[EOS]} for termination. During inference, the learned \texttt{[EOS]} signal enables early stopping, while the learned \texttt{[VOID]} signal guides adaptive response canvas expansion. On Dream-7B-Instruct, VoidPadding improves the block-size-averaged four-task mean across mathematical reasoning and code generation benchmarks by \(+17.84\) points over the original model and \(+6.95\) points over RainbowPadding, while reducing decoding NFE by 55.7\% on average. Code is available at https://github.com/Haru-LCY/VoidPadding.

## 1. 论文的核心问题与整体含义（研究动机和背景）

掩码扩散语言模型（MDLM）通过双向注意力对预分配的响应画布进行去噪来生成文本。在指令微调（IT）中，为了批处理效率，序列通常被填充到相同长度。现有MDLM继承了自回归语言模型的惯用做法——使用重复的`[EOS]`标记作为填充符。这导致`[EOS]`同时扮演两个角色：**语义终止符**（标记回答结束）和**填充符**（占位使序列等长）。论文指出，这种角色混淆是大块解码（large-block decoding）下“`[EOS]`溢出”现象的根本原因：当模型在画布尾部生成高置信度的`[EOS]`后，由于双向注意力的存在，这些`[EOS]`会作为上下文进一步增大附近位置的`[EOS]`概率，导致模型过早停止输出短回答、重复生成`[EOS]`，严重损害生成质量和效率。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
解耦`[EOS]`的双重角色：引入新的特殊标记`[VOID]`专门用于填充，保留`[EOS`仅用于语义终止。训练时将填充符替换为`[VOID]`，推理时禁止生成`[VOID]`，并利用学习到的`[EOS]`信号实现早停，利用`[VOID]`信号实现自适应画布扩展。

### 关键技术细节
- **训练数据构造**：对于每个prompt–response样本，将原始响应后附加一个`[EOS]`，然后使用`[VOID]`填充至batch内的最大序列长度，得到 `y = r ⊕ [EOS] ⊕ [VOID]*`。标准MDLM训练过程（掩码、损失计算）应用于该序列。
- **推理设计**：
  - **禁止生成`[VOID]`**：在每一步解码前将`[VOID]`的logit设为`-∞`，防止`[VOID]`成为可见输出。
  - **`[EOS]`终止策略**：一旦左侧所有可见位置（包括prompt和已生成的非`[EOS]`响应）均被还原且最早出现的`[EOS]`之前无掩码，则停止解码；丢弃`[EOS]`及之后的内容。与标准MDLM“直到所有掩码填充完毕才停止”的策略（VanillaStopping）相比，可显著减少前向传播次数（NFE）。
- **VoidExpansion自适应画布扩展**：利用`[VOID]`的长度信号。对当前画布长度`ℓ`进行一次前向传播，计算尾部窗口的平均非`[VOID]`概率`s_{nonvoid}`以及`[VOID]`概率与最大非`[VOID]`概率的差距`s_{gap}`。当`s_{nonvoid} > τ_{nonvoid}`或`s_{gap} < τ_{gap}`时，认为画布不足，以块大小`B`递增扩展，直到满足条件或达到最大长度`L_{max}`。该过程在正式去噪前完成，仅需额外前向传播（通常很少）。

### 公式（文字说明）
论文定义了`p^{void}_i` = 模型预测位置i为`[VOID]`的概率，`p^{nonvoid}_i` = 模型预测非`[VOID]`的最大概率。尾部窗口平均`s_{nonvoid}`和差距`s_{gap}`作为决定画布是否足够的统计量。

## 6. 论文的主要结论与发现

1. **`[EOS]`溢出是训练–推理角色不匹配引起**：双向注意力使填充`[EOS]`成为去噪目标，学习到长度信号；大块解码时该信号转化为溢出。
2. **VoidPadding显著提升大块鲁棒性**：固定长度B=512时，原始LLaDA GSM8K准确率从76.80（B=64）跌至28.43，而VoidPadding保持约78.62；Dream提升更大（+17.84均值）。
3. **`[EOS]`终止策略高效**：相比VanillaStopping，VoidPadding减少NFE平均63.8%（LLaDA），且不牺牲甚至提升质量；而原始模型和RainbowPadding使用`[EOS]`终止反而略降。
4. **VoidExpansion优于现有推理时自适应方法**：在短画布场景下，VoidExpansion平均准确率51.34 vs DAEDAL 50.77，且NFE更低（172.10 vs 228.82），墙钟速度提升2.12倍（几何平均）。
5. **VoidPadding对超参数选择鲁棒**：阈值、初始长度、尾窗口、块大小在合理范围内变化时，性能波动很小。
6. **Base模型同样受益**：在未指令微调的基础模型上，VoidPadding避免了`[EOS]`填充导致的灾难性溢出，RainbowPadding也有改进但VoidPadding更稳定。

- **消融详实**：对VoidExpansion的四个超参数逐因素变化并记录性能范围，展示鲁棒性。

## 8. 不足与局限

- **仅使用LoRA微调**：所有VoidPadding训练均基于LoRA（rank=32），未验证全参数微调时的表现，泛化性可能有局限。
- **仅测试中等规模模型**：实验限于8B/7B级别，未见更大规模（如70B）验证。推理效率优势在大模型上可能更显著，但尚未验证。
- **任务覆盖有限**：主要聚焦数学和代码生成，未包括开放域对话、长文本生成、机器翻译等任务，通用性未充分验证。
- **解释性有限**：`[VOID]`信号为何能用于画布扩展仅通过实验展示，缺乏理论保证；VoidExpansion的阈值需要调参（尽管鲁棒性较好），可能在不同领域需重新调整。
- **额外计算开销**：VoidExpansion在去噪前需多次前向传播（但通常次数很少），且实验显示NFE低于DAEDAL，但未与不扩展的固定画布设置比较墙钟总时间。
- **未讨论语种通用性**：方法基于英文标记，但对非英文语言中特殊标记的适用性未讨论。
- **潜在风险**：若特定任务需要`[EOS]`出现在响应中部（如多轮对话中的分界），VoidPadding的`[EOS]`终止策略可能过早截断，论文未处理此类边缘情况。
- **未与近期其他MDLM加速方法对比**：如蒸馏、KV缓存等，但该工作聚焦于填充角色本身，是正交方向。
